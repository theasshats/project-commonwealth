# CLAUDE.md — Project Commonwealth

A **cooperative PvPvE Create pack** for **1.21.1 / NeoForge**, designed for small-group co-op play (~10 on a server). It's broad on purpose — food, structures, magic, colonies, mobs, decoration, vehicles — but **scarcity-driven**: ores are scarce and regional, nobody can do everything alone, so specializing and trading happen on their own (an **emergent** economy, not a forced one). Built around Create, including **Create Aeronautics** (airships/planes/vehicles).

**Design north star — the systems loop; every mod earns its place.** The pack is one causal loop (inspired by the game *Eco*), **not** a flat "five systems" list (that framing is retired): **scarcity → pressure → production → economy**, and back. **`docs/SYSTEMS.md` is the canonical structural model** — read it. A mod anchors to one of the systems: **Create** (the tech spine — a *producer*), **magic** (Ars/Iron's/Occultism — a *producer* with locked exclusives), **survival** (the *pressure* that drives demand; **seasons** is a driver feeding it, not its own system), or the **economy** (the distribution stage — it moves **value** via Numismatics/Trading Floor/bounties **and matter** via its **logistics/aeronautics** arm; **load-bearing**, not optional flavor). Under them sits **scarcity** (regional ore-gen) as the foundation, plus two cross-cutting production routes — **MineColonies** (cheaper basics + locked exclusives, across tech *and* magic) and **boss drops** (high-tier components) — that keep the producers non-overlapping so trade is necessary. **Aeronautics/logistics** is the economy's physical movement arm (its own flagship build milestone, folded into the Economy pillar), not a separate system. A mod justifies itself by feeding one of these — or it gets cut. It doesn't have to route through Create; it just has to *belong* somewhere — and **the target is that every mod ties into at least *two* systems.** One is the floor; two is the goal, so a one-system mod is a candidate for a *second* weave, not a finished job — weaving its craftables into a second system (through Create even as decoration, or as a sellable reagent, a magic catalyst, a crop that feeds a machine) is a *plus*, not a violation. Create is the spine, not a tollgate everything must pass through. Concretely, the **recipe graph should read as one or two cohesive webs, not many isolated clusters** — measure it with `tools/connectivity/` (see `docs/CONNECTIVITY.md`). **Cost model:** basics ~3×; higher tiers gain depth through *more steps* (GregTech-style), not multiplied cost. Complex tech unlocks via **MineColonies or boss drops** (KubeJS-gated; lock list defined incrementally per pillar — #220), and freshness comes from curated content updates, not world resets. For any add or change, ask: **what system does this anchor to — and, since one is the floor, what's its *second*?** (Recipe-change philosophy — balance > theme > cost — lives in `docs/RECIPES.md`.) The why lives in `docs/DESIGN.md`; the structural model in `docs/SYSTEMS.md`; the keep/cut rubric in `docs/CURATION.md` (the older modlist-audit snapshot is archived at `docs/archive/MODLIST-AUDIT.md`).

Distributed as **packwiz manifests, not jars**: the repo holds one `.pw.toml` per mod (download URL + hash), and CI builds a Prism installer zip on release (the `.mrpack` was dropped — #73). **Public release, not a private friend-group build.** The maintainers (Xela112233 + zagwar) decided the build-out is worth publishing — it's now **distributed and advertised on the open internet**, for anyone to install, not just a known crew. Still **pre-alpha**, but that public footing raises the bar: **player-facing polish, stability, broad-hardware support (varied RAM/GPUs), and clear setup docs carry real weight now** — weigh changes with a stranger on unknown hardware in mind, not just the home crew. (Earlier docs that call it a "friend-group pack" predate this decision; the *design* still targets small co-op groups, but the *audience and distribution* are now public.)

## Where you're running: Claude Code on the web

You're in an **Anthropic-managed cloud sandbox** with this repo cloned into it — **not** the maintainer's Windows machine and **not** the game server. This shapes everything:

- **Your output is a branch + PR.** Don't commit to `main`. Branch names follow the version convention below.
- **You can't reach the live server.** `ishimura` is a home box on a residential LAN; this sandbox has no path to it. Anything labeled "server" here means **editing files**, never running them — the real deploy is done by hand on the box.
- **Don't run the editor binary.** `tools/pcmc-edit.exe` is a Windows GUI app; it can't run here. Work on its Go source instead (see below).
- **Network is gated.** packwiz fetches from Modrinth/CurseForge and builds need internet. If the session's environment doesn't allow those domains, `packwiz refresh` / `add` / hash steps fail. (Maintainer: when a task touches packwiz or builds, give the session access to at least `api.modrinth.com`, `cdn.modrinth.com`, `*.forgecdn.net`, and `github.com` — or full access.)
- **You can't run the game — know when to ask for a runtime dump.** The committed digest (`tools/mod-data/`) records each recipe as an ingredient **ID set only** — no grid layout, no keys, no counts, no NBT. That's enough to *read* the web (connectivity, curation, "what touches what") but **not enough to safely re-author** a `sequenced_assembly`/`mechanical_crafting` recipe, or to verify what a conditional/compat recipe actually registers at runtime. When a task needs that fidelity, **stop and ask the maintainer for a runtime dump** instead of guessing grids: they copy `tools/recipe-dump/dump_recipes.js` into a real instance, boot, `/reload`, and send back the JSON (method + namespace filter in `tools/recipe-dump/README.md`). Dump outputs are **local working files — never commit them** (verbatim mod content, licensing-sensitive in a public repo; gitignored). Rule of thumb: *reading* recipes → digest suffices; *writing* a complex recipe or debugging "missing at runtime" → dump first.
- **Trust `git ls-remote origin`, not the sandbox's local refs, for branch state.** The clone fetches a narrow refspec, and a local ref (even `main`) can point somewhere stale or onto another branch's lineage — a session was misled this way into measuring the wrong tree. Before reasoning about "what's on main / which branches are live," run `git ls-remote --heads origin` and fetch what you actually need.

- `mods/*.pw.toml` — one manifest per mod (URL + hash, no jar bundled).

- `scripts/` — build helpers.
- `tools/pcmc-edit.exe` — editor binary, **CI-built; never hand-edit.** `tools/editor-src/` — its Go source.
- `tools/mod-data/` — the **offline facts digest** (per-jar IDs/tags/recipe+loot ID-sets), the ground truth agents read instead of jars. **Machine-maintained mirror of `mods/*.pw.toml` — never hand-edit**: `ground-truth.yml` regenerates it (fetch → extract → prune → guard) and a hand-edit is overwritten on the next run. Invariant since #312: digest ⟷ manifests reconcile 1:1 (+ `bundled-in:` jar-in-jar entries), so a *missing* digest always means *broken*, and the guard turns drift red on the PR. Pipeline details: `tools/README.md`.
- `tools/connectivity/` — the connectivity tooling (recipe + mob-drop graph, CLI report + offline viz; renamed from `tools/recipe-graph/` in #129). Measure with `python3 tools/connectivity/connectivity.py`; math in `docs/CONNECTIVITY.md`.
- `tools/recipe-dump/` — the manual runtime-dump tool (see "know when to ask for a runtime dump" above). The tool is committed; its **outputs never are**.
- The **player-facing website** lives in its **own repo** now — **`theasshats/pcmc-site`** (split out of this repo's former `site/` dir): a self-contained Go binary serving an embedded static site plus `/api/status` (Minecraft ping) and `/api/release` (GitHub Releases lookup). Runs **on the ishimura box** behind the Cloudflare Tunnel + Caddy stack, deployed by hand — **not** in this repo's CI. Served at `pcmc.ishimura.xyz`. ⚠️ Its `/api/release` still reads **this** pack repo's releases (`GITHUB_REPO=theasshats/project-commonwealth`) — the download button resolves the *pack's* latest installer, not the site binary's. The pack repo no longer builds or tests the site (`pr-checks.yml`'s Go job is editor-only).
- `docs/` — `DESIGN.md` (the why), `EDITING.md`, `PRISM-SETUP.md`.

## packwiz conventions

- Mods are added by **slug** (Modrinth or CurseForge). There's no search UI — a slug is required.
- **New mods are forced to `side = "both"`.** Modrinth's per-version side metadata is frequently wrong (mods marked client-only that the server actually needs, and vice-versa); trusting it causes silent breakage at install time. Only narrow a mod's side later, **with a written reason in the manifest** — and remember `side = "server"` also means **single-player never gets the mod** (SP runs everything client-side): `smarterfarmers` was narrowed reason-less in PR #88 and farmers silently stopped replanting in SP worlds until #312 caught it. A side="server" mod is also invisible to client-side fetches — the digest pipeline backfills it, and a **non-zero backfill count in a `ground-truth.yml` run log is the tell** that someone narrowed a side.
- **packwiz-installer is an updater, not a downloader** — it reconciles the target directory to the requested `side`, *deleting* files outside the new target set. Never run it twice with different sides into one directory expecting a union (the second pass deletes the first's side-exclusive jars); fetch the remainder directly from manifest URLs instead, as `ground-truth.yml`'s backfill does.
- After any manifest change, run `packwiz refresh` to rebuild `index.toml`.
- **A `packwiz` binary is vendored at `tools/packwiz`** (linux/amd64, `.packwizignore`'d so it never enters the pack). Use it directly in agent sessions — `./tools/packwiz refresh`, `./tools/packwiz modrinth add <slug>` — instead of `go install`ing it each time. (Rebuild with `GOBIN=$PWD/tools go install github.com/packwiz/packwiz@latest` if it ever needs bumping.)
- **Pin gotcha:** when hand-editing a `.pw.toml`, `pin = true` must sit at the **file top level**, not under a `[update.modrinth]` (or any) section — a key after a `[section]` header belongs to that section, so a misplaced pin is silently ignored by `packwiz update`. The editor writes this correctly; hand-edits must not get it wrong.

## The editor (`tools/editor-src/`)

A custom Go web-UI app wrapping routine packwiz ops (add/remove/pin/set-version/batch-add/build). The committed `.exe` is rebuilt by `build-editor.yml` on every push to `main` touching `tools/editor-src/**` (or via manual `workflow_dispatch` — e.g. to build the `.exe` on a PR branch for playtesting) — it cross-compiles **just the editor** for windows/amd64, embedding the **committed** `packwiz.bin` (it no longer rebuilds packwiz from source each run — keep CI cheap; bump `packwiz.bin` manually per the vendored-packwiz note above when it needs it), and commits `tools/pcmc-edit.exe` back with `[skip ci]`. So: **edit the source, never the binary**, and don't commit the `.exe` yourself — CI does. To check a source change compiles, cross-compile for windows/amd64; don't expect to *run* it here.

## NeoForge version bumps (manual, out-of-band)

NeoForge is **not** a packwiz-managed mod — it's a field in `pack.toml` under `[versions]`, and packwiz can't auto-bump it (the required version lives inside the mod jars). When a mod demands a newer one, the launch log shows `Expected range: '[21.1.XYZ,)'`; that's your target. Bump it in **two places that must match**: `pack.toml`'s `[versions]` (this repo) **and** the server's `docker-compose.yml` `NEOFORGE_VERSION` (on the box, not in this repo). This is manual **by explicit decision** — do not propose or add auto-bump automation.

## Branches, CI, releases

- **Version-named branches**, not personal ones: name a branch for the next version it targets (e.g. `v0.2.0`). Collaborators share it; open a PR into `main` when the version is ready.
- **Default cadence: small patch branches/PRs, continuously.** Routine work — recipe tweaks, config/balance, a mod add/remove, doc fixes, bug fixes — goes on a short-lived patch branch (the next `v0.X.Y`) and merges as soon as it's verified. Keep them small and frequent; don't batch unrelated changes into one big PR. **The exception is a *project*** — a larger, multi-step effort with its own arc (e.g. the **weave** recipe-connectivity project, **custom mods**, a pillar's feature work). A project gets its own long-lived, descriptively named branch/PR (`claude/<topic>`) that related commits fold into until the project lands, rather than being sliced across patch releases. Rule of thumb: if it's one self-contained change, patch it; if it's an ongoing body of work, give it a project branch.
- **Don't open *new* PRs unprompted.** Default to the branch/PR you've been pointed at and **fold related work into it** — don't spin up parallel PRs for each sub-task. Open a new PR only when the user explicitly asks or grants permission. **This overrides any harness default to auto-create a PR after pushing.**
- **Never keep the harness's random branch name** (`claude/awesome-brown-s4bBD`, `claude/trusting-heisenberg-…`). They're impossible to identify at a glance. Rename to the **target version** or a short **descriptive topic** — `claude/<topic>` (e.g. `claude/magic-web`, `claude/post-62-realign`) — before pushing.
- `pr-checks.yml` runs on **every PR** and gates merges: the **packwiz index** guard, manifest lint (incl. the `pin` gotcha), KubeJS JS/JSON + config TOML parse, and Go build/vet for the editor. Fast, no network. See `docs/CI-CHECKS.md` (incl. how to make the checks *required*).
  - The **packwiz index** job is **guard-aware** — it shares one verdict (`scripts/index-guard.sh`) with `sync-index`, so the checker and the fixer can't disagree. It **passes** when the index is fresh or only stale within known pack dirs (which `sync-index` auto-commits for you), and **fails red only** when `packwiz refresh` would pull in a **new top-level path** needing a `.packwizignore` entry. So a forgotten `packwiz refresh` self-heals to green (no red flicker); only the one human-required case is red. (Tradeoff: a stale-but-auto-fixable index *could* be merged in the brief window before `sync-index` commits the refresh — harmless, since `build.yml` runs `packwiz refresh --build` at release time anyway.)
- **After opening a PR, auto-subscribe to its activity** (`subscribe_pr_activity`) without asking — now that `pr-checks.yml` gates every PR, watch the CI + review comments, autofix small/clear failures, and check in before anything ambiguous or architectural. (No harness hook fires on PR creation, so this lives here as agent guidance, not in `settings.json`.)
- **Green CI ≠ it loads. Standardize a playtest on every gameplay PR.** `pr-checks.yml` is **static and parse-only** — it confirms JSON/TOML *parse* and Go *builds*, but it **cannot** validate datapack/worldgen schemas, mod compat, or runtime behavior, and the sandbox **can't launch the game**. Schema-valid-but-wrong content sails through and crashes at load (real example: a GTMOGS vein layer keyed `target` instead of the required `targets` — valid JSON, passed CI, crashes world creation). So: **any PR that changes pack content** (mods, `kubejs/`, worldgen/biome data, `config/`, recipes, NeoForge bump) **must carry a `## Playtest` checklist in its description** and **stay a draft until a human boots a fresh world and ticks it.** The checklist is PR-specific but always covers the load-bearing basics: client boots to menu, a fresh world creates without errors, the changed feature actually works in-game, and the server jar starts (since some mods are client-only). When you open such a PR, **write the checklist** (tailored to what changed) and say plainly in your summary that CI passing does **not** mean it's been verified. **This applies to *whatever* PR your change lands in — always add your change's playtest steps to that PR.** If you fold work into an existing or integration PR (e.g. a `vX.Y.Z` branch) rather than opening your own, **add a playtest sub-block for your change to that PR's `## Playtest` section** — never let a content change ride into a PR without its own playtest steps. A change is not "done" until its playtest items exist on the PR it ships in.
- `sync-index.yml` is the **single owner** of regenerating the packwiz-generated `index.toml`/`pack.toml` on a PR (it folded together the old `resolve-conflicts.yml` + `refresh-index.yml`, which overlapped and could race — #127; one workflow + per-PR concurrency can't race itself). On push to `main`, PR open/reopen/**synchronize**, or dispatch it: merges `main` if the branch is behind and rebuilds the index from the combined content (auto-resolving **only** regenerable conflicts — `index.toml` always, and `pack.toml` when it differs **solely in the `[index]` hash**, or **solely in the version line with this branch's version *ahead* of main's**: a **version branch folding in the current release while keeping its own higher version**, so in-progress versions like `v0.6.0` are always tested on the latest shipped `0.5.x` content; it **bails to a human** on real content conflicts (**except `docs/PATCHNOTES.md`**, which a `.gitattributes` `merge=union` driver auto-merges — both sides' changelog sections are kept, so release folds don't conflict on it), and handles a branch whose version is *below* main's as **superseded** — a `0.5.x` branch left open after `0.6.0` shipped: an **empty staged slot** (only the auto-staged bump, no human commits) is **auto-closed and its branch deleted** (the correctly-numbered next patch is already staged), while one carrying **real work** is **flagged** to close/rebase rather than have the release mislabelled into it); and, for a branch that's merely **stale** (edited a tracked file, forgot `packwiz refresh`), auto-commits with `[skip ci]` as long as the index changes stay within **known pack dirs** (the `scripts/instance-dirs.txt` set + `mods/` + already-indexed prefixes) — so a new mod/config/`tacz` zip rolls over — bailing **only** when a file enters under a **new top-level path** (a dir that should be `.packwizignore`d), which is the narrowed `.packwizignore` guard. **Manual fix when it bails:** merge `main`, then either `packwiz refresh` + commit, or — for a docs-only PR — `git checkout origin/main -- index.toml pack.toml`. Steps in `docs/CI-CHECKS.md`. The push retry is the shared `push-with-retry` composite action. Its checkout/push uses **`secrets.SYNC_PAT`** (a fine-grained PAT or GitHub App token) when set, so the fold/refresh push triggers a **normal `pr-checks` run that displays on the PR** (the green tick + required-check status); it falls back to `GITHUB_TOKEN`, whose pushes **don't** re-trigger `pull_request` CI (anti-recursion) — in which case a bot-folded commit shows **no** checks on the PR until a human pushes. (`workflow_dispatch` can't fill this gap: those check runs exist on the commit but GitHub doesn't surface them in the PR's merge box, so the PAT is the only way to get a *displayed* green tick. A GitHub App is the more locked-down version of the same — issue tracked for 1.0.0.) `release.yml`'s `open-next` uses the same secret so staged PRs show checks too. The old browser **dispatch mod-management workflows (add-mod / remove-mod / update-mods / compute-hash / refresh) were retired** — the editor does all of it now (#127).
- `build.yml` builds the Prism installer zip (the `pack.toml` URL baked in at build time) and attaches it to a release. It's a **reusable workflow** (`workflow_call`, invoked by `release.yml`), and also runs on a hand-made `release:published` or manual `workflow_dispatch` (test build — **builds the branch it's dispatched from, not `main`**, and uploads the installer zip as a downloadable **workflow artifact** named `prism-installer` rather than a release, 14-day retention, so a version branch like `v0.7.0` can be playtested before it merges; the updated `build.yml` must be present *on that branch* for the dispatch to use it). No caching — the build is ~30s. (No `.mrpack`: it bloated to ~200 MB and nobody used it — dropped per #73.)
- `release.yml` **auto-cuts a release** on push to `main` (i.e. when a version PR merges). It reads `version` from `pack.toml` and, **only if tag `vX.Y.Z` doesn't already exist**, calls `build.yml` to create the tag + release with the zip attached. The tag-exists guard makes it idempotent — ordinary main pushes (docs, the `sync-index` `[skip ci]` auto-commits) don't change the version, so they're no-ops. **Version-first**: `pack.toml` drives the tag, so the old "sync pack.toml from the tag" step in `build.yml` was removed.
- `build-editor.yml` runs on `tools/editor-src/**` changes (see above).
- `ground-truth.yml` keeps the **mod-data facts digest** (`tools/mod-data/` — IDs/tags/biome-modifiers only, license-safe) an exact mirror of the manifests, **self-pruning since #312**: fetch (client bootstrap + hash-verified backfill of any side-narrowed jar, exactly once per mod) → extract (`scripts/extract-mod-data.sh`, write-only) → **prune** (`scripts/prune-mod-data.sh` — deletes digests of cut/superseded jars; keeps `bundled-in:` entries while their outer jar is installed) → **guard** (`CHECK=1`, same script/verdict — **fails the job on any digest⟷manifest drift**, so missing always means broken) → commit-back. Runs on `mods/**`/`pack.toml` changes (push to `main` **and** PRs incl. drafts — committed back to the PR branch), a weekly cron backstop, or dispatch; invariants hold **per-branch**. No loop (it doesn't watch `tools/mod-data/**`); its commit feeds `connectivity-web.yml`. A red guard is a question for a human (fetch failure, broken jar, genuine drift) — never "run the fixer"; the fixer already ran.
- `connectivity-web.yml` rebuilds the connectivity viz (`tools/connectivity/connectivity-web.html`) and refreshes the data block in `docs/CONNECTIVITY.md`. Runs on push to `main` touching recipes / the digest / the connectivity tooling, or dispatch; commits back with `[skip ci]` (same anti-loop pattern as `build-editor.yml`).
- Cutting a release: **bump `version` in `pack.toml` (+ a `docs/PATCHNOTES.md` entry) in the PR → merge to `main`**. `release.yml` tags `v0.X.Y` and publishes the assets in ~1 min. No manual tagging. (Forgetting the bump = no release; the version is the trigger, so don't miss it.)
- **The GitHub release body is pulled from `docs/PATCHNOTES.md`** — `build.yml` extracts the `## X.Y.Z …` section for the version being released (up to the next `## `). So **keep `PATCHNOTES.md` current on the release branch for any change you make there**: whatever that section says when the branch merges is what ships as the public release notes. If it's missing or still the auto-staged `— TBD` stub, the release still cuts but its body is a "notes not finalized" placeholder (and a CI warning fires) — so the failure mode is ugly notes, never a blocked release. ⚠️ **Add a change's note on the version branch it ships with — don't pre-add a future version's `## X.Y.Z` heading on `main`**: the staged version branch already carries that heading, and the `merge=union` driver keeps *both* sides' lines on the fold, so a heading added on `main` too becomes a duplicate section. If your player-facing change merges to `main` directly (e.g. a tooling PR with a small content edge), flag the note for the next version branch instead of editing `PATCHNOTES.md` on `main`.
- **Every issue maps to a milestone**, on an **odd/even release cadence** (canonical map in `docs/ROADMAP.md`; rationale archived at `docs/archive/RELEASE-CADENCE.md`): **odd = feature/integration + the curation "thunderdome"** (curation by merit), **even = stabilization** (perf/bug/balance, *no new mods*). One pillar per odd version — `v0.7.0` Create spine → `v0.9.0` Survival → `v0.11.0` Magic & MineColonies → `v0.13.0` Economy & logistics (incl. aeronautics) → `v0.15.0` Polish + weave review — each trailed by a stabilization even; `v0.6.0` (Foundation) and `v1.0.0` (even, feature-frozen perf patch) bookend, plus `📋 Backlog / living`. The thunderdome runs on the **odd** versions; perf-pruning on the evens. Each milestone carries a one-line **goal** (its exit criterion); the full map is in `docs/ROADMAP.md` ("Milestones — the road to 1.0"). **Any new issue must be filed under a milestone** — its pillar's odd version, the stabilization even that follows, or `Backlog / living` for ongoing/not-yet-scheduled work (the backlog is expected to grow well past today's count, so it's the holding pen that keeps everything triaged). An unmilestoned issue is untriaged. Milestones are created/edited on GitHub, not in this repo.
  - **Close issues on *stage*, not on merge** (`docs/TRIAGE.md` §5): close an issue as `completed` when its fix lands on the version branch, so the milestone bar climbs *as the version is built* — that bar is the at-a-glance "how much of this version is done?" gauge. Keep `playtest-blocked` on the closed issue until it's verified in-game (drop it once playtest passes; a failed playtest reopens). Milestone 100% = all work *implemented*; the hard "shipped" signal is the `vX.Y.Z` release tag (`release.yml`), i.e. 100% **and** no `playtest-blocked` left **and** the tag cut.
  - **Milestone *numbers*** (the GitHub API needs the number, not the title, when filing/moving an issue — so they're recorded here to save a lookup; verify against `/milestones` if a mismatch is suspected): **10** = `v0.6.0 — Foundation`, **11** = `v0.7.0 — Create spine`, **12** = `v0.8.0 — Stabilization I`, **13** = `v0.9.0 — Survival`, **14** = `v0.10.0 — Stabilization II`, **15** = `v0.11.0 — Magic & MineColonies`, **16** = `v0.12.0 — Stabilization III`, **17** = `v0.13.0 — Economy & logistics`, **18** = `v0.14.0 — Stabilization IV`, **19** = `v0.15.0 — Polish & site`, **20** = `v1.0.0 — Release [NFR]`, **21** = `📋 Backlog / living`. _(Milestones 13/15/17 have been retitled twice, both times via `scripts/restructure-milestones.sh`: first when the model dropped the five-systems framing, then when the pillar order was rebuilt to walk the loop — Survival up to v0.9.0 (pressure before distribution), MineColonies joining Magic at v0.11.0 (the two non-Create production routes, one owner each), Economy & logistics last at v0.13.0 (it's mostly whole-pack balance, which needs the other pillars to exist).)_

## Mod / gameplay gotchas

- `aeronauticscompat` (CurseForge) is **required, not optional** — it patches third-party mods to behave on physics-simulated ships (Etched, Cobblemon, turrets, sleeping, chains, …). Keep it in the pack.
- **Recipe viewer is JEI, not EMI.** EMI **cannot render Create's processing recipes** (pressing/mixing/deploying/sequenced-assembly) on 1.21.1 — that hid the entire Create spine (even `create:iron_sheet` showed "no recipe"), which is fatal for a Create pack. Switched to **JEI** (+ **JER** for loot/worldgen/trades, **JEED** for effects) in the recipe-overhaul work. **Don't re-add EMI.**
- **`galosphere:silver_ingot` is actually PALLADIUM** — Galosphere renamed its metal silver→palladium but kept the legacy item *id*. It's tagged `c:ingots/palladium` (its real material), **not** `c:ingots/silver`; the real silver is `occultism`. ⚠️ **Never unify `galosphere:*silver*` as silver** (e.g. an almost-unified `priority_override` on `c:ingots/silver` → galosphere is wrong, and would silently merge palladium into silver). Watch for this on auto-updates that re-touch unification.
- `itemphysic` and `FoamFix` are **client-only — never on the server** (they crash it).
- **Inventory sorting is Inventory Essentials — IPN was removed in v0.7.1 (#257).** Inventory Profiles Next + `libipn` + `invtweaks-emu-for-ipn` were swapped for the single-mod `inventory-essentials` (sorting + essential tweaks, no deps), which also retired the #3 empty-slot-arrow KubeJS texture override (`kubejs/assets/inventoryprofilesnext/` — deleted). `kotlin-for-forge` STAYS (fzzy_config, kambrik, Slice & Dice need it). If IPN is ever re-added, remember #3: its magenta empty-slot arrow has no config toggle and needs the hand-cleared `overlay_new.png` override (history in #3/#257).
- **Sodium runs *only* via the Sable-compat bridge — never add it bare, never add Embeddium.** Bare Sodium 0.6.9+ on NeoForge 1.21.1 turns **all world textures invisible** when Create: Aeronautics is present (Sodium [#3606](https://github.com/CaffeineMC/sodium/issues/3606), still open upstream; no `sodium-mixins.properties`/render-mode workaround). Aeronautics is **required** (so is `aeronauticscompat`) and hard-depends on **Sable → Veil**. The pack works around #3606 with a **community bridge** (added v0.6.0, #98): **Sodium `0.8.12-alpha.4` + `sable-new-sodium-compat` 1.0.0 + explicit `veil-lib` 4.1.3 + Sable 1.2.2** — the compat backports Sable's new-Sodium support to 1.2.2 and makes Aeronautics render. **All three additions are pinned** as a matched set; don't `packwiz update` them apart, and if you bump one, re-verify the quartet boots with world textures intact. (Embeddium is *incompatible with Veil* outright — still off the table.) ⚠️ Without this bridge the F3 line `[Fabric] Active renderer: IndigoRenderer` means no chunk optimizer is active. **Why Sodium at all:** #98 traced the ~37-FPS spawn to a **CPU-bound** frame — F3 showed `GPU: 49%` (GPU half-idle), looking at open sky reclaimed the frames, so the cost was the **vanilla chunk renderer submitting terrain on the render thread**, exactly what Sodium replaces. Draw-side mods (ImmediatelyFast + EntityCulling) were trialed first and **reverted — no gain** (the cost was terrain, not entities/TER). Earlier docs wrongly claimed "the pack uses Sodium proper" — it never did until this bridge. **Status: a TRIAL pending in-game verification** (the #3606 failure mode — invisible world textures — is what to watch); if it regresses, revert the additions and wait on upstream #3606. ⚠️ **The bridge forces *alpha* Sodium (`[0.8.12-alpha.2,)`) — it will NOT load on stable 0.6.x — and alpha Sodium's renderer rewrite breaks ecosystem mods' Sodium-compat mixins.** That made the trial a cascade, now resolved by a matched set you must keep together: **NeoForge `21.1.233`** (Quark/Towns&Towers floor from the v0.6.0 mod update), **Supplementaries `3.6.7` + Moonlight `3.0.14`** (older Supplementaries' fluid mixin crashed; 3.6.7 needs Moonlight ≥2.30.0), **Create More: Package Couriers `2.3.0`** (Supplementaries 3.6.7 moved `RegisterFireBehaviorsEvent` `api.neoforge`→`api.platform`), and **`mixin-helper`** shipping `config/mixinhelper.json` that blacklists `net.mehvahdjukaar.supplementaries.mixins.neoforge.compat.CompatSodiumFluidRendererMixin` — Supplementaries 3.6.7 *still* crashes against alpha Sodium at world-render (`Scanned 0 target(s)`, upstream [Supplementaries #2011](https://github.com/MehVahdJukaar/Supplementaries/issues/2011) open), and that blacklist is the only fix (cosmetic cost: Lumisene fluid loses its emissive glow under Sodium). **Don't remove `mixin-helper`, that config line, or downgrade Supplementaries/Moonlight while Sodium is in** — any of those re-breaks world-load. Revisit/remove the blacklist when Supplementaries ships real Sodium-0.8 support (#195).
- **Iris/Oculus shaders break Aeronautics ships visually** — known upstream bug, no fix. If ships look wrong, the answer is "disable shaders," not debugging shader compat.
- **Boot-error cascades: the first error is the real one.** The rest are dependents failing because their dependency failed; fix the first and the cascade usually clears. Missing-dependency errors are usually packwiz not auto-pulling a dep that isn't jar-in-jar — add it manually.
- **Create Big Cannons is version-locked to its addon — keep CBC at 5.11.3.** The **Advanced Technologies addon** (`cbc_at`, currently `0.1.4c`, the newest 1.21.1 build) is built against **CBC 5.11.3** and calls `PartialBlockDamageManager.damageBlock` with a signature CBC **changed after 5.11.3**. On 5.11.5 a heavy autocannon AP round striking a block throws `NoSuchMethodError` and **crashes the server** (`Ticking entity` / `cbc_at:ha_ap_projectile`). There's no forward fix (0.1.4c is the latest addon), so the alignment is CBC pinned to **5.11.3**. ⚠️ **Don't let `packwiz update` bump `create-big-cannons` past 5.11.3 while `cbc_at` is 0.1.4c** — it re-breaks world-runtime. Re-pair only when a newer `cbc_at` targets a newer CBC. (Fixed v0.6.2; the v0.6.2 release branch had drifted to 5.11.5 while `main` was already 5.11.3.)
- **GTMOGS prospecting is *right-click*, not break — and the map layer is render-only.** Verified by decompiling `gtmogs-1.21.1-1.0.6`: `CommonEventListener.onRightClick` (`PlayerInteractEvent$RightClickBlock`) is the **only** prospect trigger — there's no block-break handler and no prospector item. Right-clicking a vein ore scans within `oreBlockProspectRange` (default 24) → `Prospected N new veins!`. The Xaero's/FTB Chunks/JourneyMap integrations only **draw** already-prospected veins; they don't trigger prospecting. So "prospecting broken" is almost never the map mod — check that players are right-clicking, not breaking. Config is **YAML** at `config/gtmogs.yaml` (toma `configuration` lib); `borderColor` is **ARGB** and defaults to transparent `#00000000` (invisible marker outlines) — the pack ships it as opaque hot pink `#FFFF1493`. ⚠️ The lib's YAML parser is **hand-rolled, not real YAML** (verified in `configuration` 3.1.0's `YamlFormat.readFile`): it strips only *whole-line* `#` comments and takes everything after `key: ` verbatim, so the **unquoted** `borderColor: #FFFF1493` is read correctly — standard YAML tooling misparses that line as null-plus-comment, and "fixing" it with quotes would put literal quote characters in the value and break it. Don't quote it, and don't trust YAML linters on this file. Player-facing details live in `wiki/prospecting.md`.

## Server deploy seam (reference — runs on the box, not here)

The server uses `TYPE=NEOFORGE` and pulls mods from the live `pack.toml` via packwiz-installer at startup, so "deploy a new version" = restart the container, which re-pulls. The only repo-side server task is the NeoForge `[versions]` bump above. Performance (TPS) is profiled with **spark** on the live server — and **measure, don't analogize**: old-pack tuning lessons (e.g. NTNH's 12G-vs-16G) were 1.7.10 / Java 8 and don't carry to 1.21.1 / Java 21 / G1GC.

- **The box's deploy config lives in its own repo, `theasshats/pcmc-server`** (`docker-compose.yml` + `auto-update.sh` + a runbook — the site-split pattern), **not** in this repo. It replaces the old split-brain deploy where NeoForge (from `.env`) and mods (floating on `main`, re-synced every container start) drifted apart on any unscripted restart — the cause of the `21.1.228`-vs-`21.1.233` crash loop. `auto-update.sh` resolves **one immutable commit** from a channel (`release` = production / `branch` = playtest, e.g. `v0.7.0`), reads `pack.toml` at that commit, and pins **both** `NEOFORGE_VERSION` and a commit-pinned `PACKWIZ_URL` into `.env` together, then gates the deploy on the container's real healthcheck with rollback to the prior commit. So the box-side NeoForge value is now **derived** from the pinned commit's `pack.toml` (the script *mirrors* the human decision — it still never bumps `pack.toml` itself), narrowing the "two places that must match" to just the repo-side `[versions]` edit. **The pack serves client *and* server from one `pack.toml` via packwiz `side` filtering — see `docs/SERVER.md`** (delivery paths + the side-metadata rule that keeps both installs healthy).

## Working style

Keep changes focused and explain them briefly. Before changing a mod's presence or a config's values, check `docs/DESIGN.md` (and `mods/README.md` for manifest conventions) — a lot of in/out decisions are deliberate and documented.

- **End-of-session docs pass — when the maintainer flags it** (e.g. "docs pass", "wrap up", "lessons learned"): sweep the docs the session's work *talks about*, not just the ones it edited. Three checks: (1) **currency** — find claims the session made stale (counts, "X is the only…", pipeline descriptions, paths) and fix them; (2) **lessons** — record what the session learned *at the point of future contact*, i.e. the doc/section a future instance will actually be reading when the trap matters (a side-narrowing lesson goes in the side rules, a CI gotcha in the workflow bullet, an agent-environment trap in the sandbox section — not a separate lessons file); (3) **agent context** — does CLAUDE.md give a future instance enough to make the right call without rediscovering it? Fold the pass into the session's open PR rather than a new one. This runs **on the maintainer's flag**, not automatically — but if a session ends with obviously stale docs and no flag, say so instead of silently leaving them.

- **Player-facing copy — chiefly `docs/PATCHNOTES.md`, plus site text — reads professionally**, the way a person would write it: no decorative emoji, sparing bold, plain declarative sentences, and no hype or filler ("the big one", "you'll notice immediately", "seamlessly", "actually"). **Use bullets where they make sense**: independent changes (fixes, additions, tweaks) get one bullet each — lead with the change, then the why/impact in the same bullet; keep prose paragraphs for narrative that doesn't enumerate (a new system's introduction, a rename). The style note at the top of `PATCHNOTES.md` is the canonical form; shipped release sections are the published record — don't restyle them retroactively. Internal docs (handoffs, ROADMAP, RECIPES, CLAUDE.md) and commit/PR text **keep their normal heavier formatting** — the bold/structure aids parsing; this rule is for player-facing copy only.
