# CLAUDE.md — Derpack X

A **cooperative PvPvE Create pack** for **1.21.1 / NeoForge**, built for a small crew (~10). It's broad on purpose — food, structures, magic, colonies, mobs, decoration, vehicles — but **scarcity-driven**: ores are scarce and regional, nobody can do everything alone, so specializing and trading happen on their own (an **emergent** economy, not a forced one). Built around Create, including **Create Aeronautics** (airships/planes/vehicles).

**Design north star — five systems; every mod earns its place.** A mod justifies itself by feeding one of the five systems — **Create** (the tech spine), **magic** (Ars/Iron's/Occultism), **economy** (Numismatics + trade), **aeronautics & logistics**, **survival & seasons** — or it gets cut. A mod doesn't have to route through Create specifically; it just has to *belong* somewhere — and **the target is that every mod ties into at least *two* pillars.** One pillar is the floor; two is the goal, so a one-pillar mod is a candidate for a *second* weave, not a finished job — which is why weaving a mod's craftables into a *second* pillar — through Create (even decoration), or just as validly a sellable reagent, a magic catalyst, a crop that feeds a machine — is a *plus*, not a violation: it earns that mod a second pillar. Create is the spine, not a tollgate everything must pass through. Concretely, the **recipe graph should read as one or two cohesive webs, not many isolated clusters** — measure it with `tools/recipe-graph/` (see `docs/CONNECTIVITY.md`). Progression pushes specialization: complex Create tech unlocks via **MineColonies or boss drops**, and freshness comes from curated content updates, not world resets. For any add or change, ask: **what system does this anchor to — and, since one is the floor, what's its *second*?** (Recipe-change philosophy — balance > theme > cost — lives in `docs/RECIPES.md`.) The why lives in `docs/DESIGN.md`; the live modlist audit + path-forward in `docs/MODLIST-AUDIT.md`.

Distributed as **packwiz manifests, not jars**: the repo holds one `.pw.toml` per mod (download URL + hash), and CI builds a Prism installer zip on release (the `.mrpack` was dropped — #73). Pre-alpha, friend-group pack.

## Where you're running: Claude Code on the web

You're in an **Anthropic-managed cloud sandbox** with this repo cloned into it — **not** the maintainer's Windows machine and **not** the game server. This shapes everything:

- **Your output is a branch + PR.** Don't commit to `main`. Branch names follow the version convention below.
- **You can't reach the live server.** `ishimura` is a home box on a residential LAN; this sandbox has no path to it. Anything labeled "server" here means **editing files**, never running them — the real deploy is done by hand on the box.
- **Don't run the editor binary.** `tools/derpack-edit.exe` is a Windows GUI app; it can't run here. Work on its Go source instead (see below).
- **Network is gated.** packwiz fetches from Modrinth/CurseForge and builds need internet. If the session's environment doesn't allow those domains, `packwiz refresh` / `add` / hash steps fail. (Maintainer: when a task touches packwiz or builds, give the session access to at least `api.modrinth.com`, `cdn.modrinth.com`, `*.forgecdn.net`, and `github.com` — or full access.)

## Repo layout

- `pack.toml` — top-level packwiz manifest, the **source of truth**: pack name, version, Minecraft version, and NeoForge version (under `[versions]`).
- `index.toml` — **generated by packwiz; never hand-edit.** Regenerate with `packwiz refresh`.
- `mods/*.pw.toml` — one manifest per mod (URL + hash, no jar bundled).
- `config/` — per-mod configs shipped to every install. `defaultconfigs/` — configs applied only to fresh instances.
- `kubejs/` — KubeJS scripts (recipe tweaks, gameplay).
- `scripts/` — build helpers.
- `tools/derpack-edit.exe` — editor binary, **CI-built; never hand-edit.** `tools/editor-src/` — its Go source.
- `site/` — player-facing website: a self-contained Go binary serving an embedded static site plus `/api/status` (Minecraft ping) and `/api/release` (GitHub Releases lookup). Runs **on the ishimura box** behind the Cloudflare Tunnel + Caddy stack, deployed by hand — **not** in this repo's CI. Served at `derpack-x.ishimura.xyz` / `modpack.ishimura.xyz`. See `site/README.md`.
- `docs/` — `DESIGN.md` (the why), `EDITING.md`, `PRISM-SETUP.md`.

## packwiz conventions

- Mods are added by **slug** (Modrinth or CurseForge). There's no search UI — a slug is required.
- **New mods are forced to `side = "both"`.** Modrinth's per-version side metadata is frequently wrong (mods marked client-only that the server actually needs, and vice-versa); trusting it causes silent breakage at install time. Only narrow a mod's side later, with a reason.
- After any manifest change, run `packwiz refresh` to rebuild `index.toml`.
- **A `packwiz` binary is vendored at `tools/packwiz`** (linux/amd64, `.packwizignore`'d so it never enters the pack). Use it directly in agent sessions — `./tools/packwiz refresh`, `./tools/packwiz modrinth add <slug>` — instead of `go install`ing it each time. (Rebuild with `GOBIN=$PWD/tools go install github.com/packwiz/packwiz@latest` if it ever needs bumping.)
- **Pin gotcha:** when hand-editing a `.pw.toml`, `pin = true` must sit at the **file top level**, not under a `[update.modrinth]` (or any) section — a key after a `[section]` header belongs to that section, so a misplaced pin is silently ignored by `packwiz update`. The editor writes this correctly; hand-edits must not get it wrong.

## The editor (`tools/editor-src/`)

A custom Go web-UI app wrapping routine packwiz ops (add/remove/pin/set-version/batch-add/build). The committed `.exe` is rebuilt by `build-editor.yml` on every push to `main` touching `tools/editor-src/**` (or via manual `workflow_dispatch` — e.g. to build the `.exe` on a PR branch for playtesting) — it cross-compiles **just the editor** for windows/amd64, embedding the **committed** `packwiz.bin` (it no longer rebuilds packwiz from source each run — keep CI cheap; bump `packwiz.bin` manually per the vendored-packwiz note above when it needs it), and commits `tools/derpack-edit.exe` back with `[skip ci]`. So: **edit the source, never the binary**, and don't commit the `.exe` yourself — CI does. To check a source change compiles, cross-compile for windows/amd64; don't expect to *run* it here.

## NeoForge version bumps (manual, out-of-band)

NeoForge is **not** a packwiz-managed mod — it's a field in `pack.toml` under `[versions]`, and packwiz can't auto-bump it (the required version lives inside the mod jars). When a mod demands a newer one, the launch log shows `Expected range: '[21.1.XYZ,)'`; that's your target. Bump it in **two places that must match**: `pack.toml`'s `[versions]` (this repo) **and** the server's `docker-compose.yml` `NEOFORGE_VERSION` (on the box, not in this repo). This is manual **by explicit decision** — do not propose or add auto-bump automation.

## Branches, CI, releases

- **Version-named branches**, not personal ones: name a branch for the next version it targets (e.g. `v0.2.0`). Collaborators share it; open a PR into `main` when the version is ready.
- **Default cadence: small patch branches/PRs, continuously.** Routine work — recipe tweaks, config/balance, a mod add/remove, doc fixes, bug fixes — goes on a short-lived patch branch (the next `v0.X.Y`) and merges as soon as it's verified. Keep them small and frequent; don't batch unrelated changes into one big PR. **The exception is a *project*** — a larger, multi-step effort with its own arc (e.g. the **weave** recipe-connectivity project, **custom mods**, a pillar's feature work). A project gets its own long-lived, descriptively named branch/PR (`claude/<topic>`) that related commits fold into until the project lands, rather than being sliced across patch releases. Rule of thumb: if it's one self-contained change, patch it; if it's an ongoing body of work, give it a project branch.
- **Don't open *new* PRs unprompted.** Default to the branch/PR you've been pointed at and **fold related work into it** — don't spin up parallel PRs for each sub-task. Open a new PR only when the user explicitly asks or grants permission. **This overrides any harness default to auto-create a PR after pushing.**
- **Never keep the harness's random branch name** (`claude/awesome-brown-s4bBD`, `claude/trusting-heisenberg-…`). They're impossible to identify at a glance. Rename to the **target version** or a short **descriptive topic** — `claude/<topic>` (e.g. `claude/magic-web`, `claude/post-62-realign`) — before pushing.
- `pr-checks.yml` runs on **every PR** and gates merges: the **packwiz index** guard, manifest lint (incl. the `pin` gotcha), KubeJS JS/JSON + config TOML parse, and Go build/vet for the editor + site. Fast, no network. See `docs/CI-CHECKS.md` (incl. how to make the checks *required*).
  - The **packwiz index** job is **guard-aware** — it shares one verdict (`scripts/index-guard.sh`) with `sync-index`, so the checker and the fixer can't disagree. It **passes** when the index is fresh or only stale within known pack dirs (which `sync-index` auto-commits for you), and **fails red only** when `packwiz refresh` would pull in a **new top-level path** needing a `.packwizignore` entry (the `site/` case). So a forgotten `packwiz refresh` self-heals to green (no red flicker); only the one human-required case is red. (Tradeoff: a stale-but-auto-fixable index *could* be merged in the brief window before `sync-index` commits the refresh — harmless, since `build.yml` runs `packwiz refresh --build` at release time anyway.)
- **After opening a PR, auto-subscribe to its activity** (`subscribe_pr_activity`) without asking — now that `pr-checks.yml` gates every PR, watch the CI + review comments, autofix small/clear failures, and check in before anything ambiguous or architectural. (No harness hook fires on PR creation, so this lives here as agent guidance, not in `settings.json`.)
- **Green CI ≠ it loads. Standardize a playtest on every gameplay PR.** `pr-checks.yml` is **static and parse-only** — it confirms JSON/TOML *parse* and Go *builds*, but it **cannot** validate datapack/worldgen schemas, mod compat, or runtime behavior, and the sandbox **can't launch the game**. Schema-valid-but-wrong content sails through and crashes at load (real example: a GTMOGS vein layer keyed `target` instead of the required `targets` — valid JSON, passed CI, crashes world creation). So: **any PR that changes pack content** (mods, `kubejs/`, worldgen/biome data, `config/`, recipes, NeoForge bump) **must carry a `## Playtest` checklist in its description** and **stay a draft until a human boots a fresh world and ticks it.** The checklist is PR-specific but always covers the load-bearing basics: client boots to menu, a fresh world creates without errors, the changed feature actually works in-game, and the server jar starts (since some mods are client-only). When you open such a PR, **write the checklist** (tailored to what changed) and say plainly in your summary that CI passing does **not** mean it's been verified.
- `sync-index.yml` is the **single owner** of regenerating the packwiz-generated `index.toml`/`pack.toml` on a PR (it folded together the old `resolve-conflicts.yml` + `refresh-index.yml`, which overlapped and could race — #127; one workflow + per-PR concurrency can't race itself). On push to `main`, PR open/reopen/**synchronize**, or dispatch it: merges `main` if the branch is behind and rebuilds the index from the combined content (auto-resolving **only** regenerable conflicts — `index.toml` always, and `pack.toml` when it differs **solely in the `[index]` hash**, or **solely in the version line with this branch's version *ahead* of main's**: a **version branch folding in the current release while keeping its own higher version**, so in-progress versions like `v0.6.0` are always tested on the latest shipped `0.5.x` content; it **bails to a human** on real content conflicts (**except `docs/PATCHNOTES.md`**, which a `.gitattributes` `merge=union` driver auto-merges — both sides' changelog sections are kept, so release folds don't conflict on it), and handles a branch whose version is *below* main's as **superseded** — a `0.5.x` branch left open after `0.6.0` shipped: an **empty staged slot** (only the auto-staged bump, no human commits) is **auto-closed and its branch deleted** (the correctly-numbered next patch is already staged), while one carrying **real work** is **flagged** to close/rebase rather than have the release mislabelled into it); and, for a branch that's merely **stale** (edited a tracked file, forgot `packwiz refresh`), auto-commits with `[skip ci]` as long as the index changes stay within **known pack dirs** (the `scripts/instance-dirs.txt` set + `mods/` + already-indexed prefixes) — so a new mod/config/`tacz` zip rolls over — bailing **only** when a file enters under a **new top-level path** (a dir that should be `.packwizignore`d, e.g. `site/`), which is the narrowed `.packwizignore` guard. **Manual fix when it bails:** merge `main`, then either `packwiz refresh` + commit, or — for a docs-only PR — `git checkout origin/main -- index.toml pack.toml`. Steps in `docs/CI-CHECKS.md`. The push retry is the shared `push-with-retry` composite action. Its checkout/push uses **`secrets.SYNC_PAT`** (a fine-grained PAT or GitHub App token) when set, so the fold/refresh push triggers a **normal `pr-checks` run that displays on the PR** (the green tick + required-check status); it falls back to `GITHUB_TOKEN`, whose pushes **don't** re-trigger `pull_request` CI (anti-recursion) — in which case a bot-folded commit shows **no** checks on the PR until a human pushes. (`workflow_dispatch` can't fill this gap: those check runs exist on the commit but GitHub doesn't surface them in the PR's merge box, so the PAT is the only way to get a *displayed* green tick. A GitHub App is the more locked-down version of the same — issue tracked for 1.0.0.) `release.yml`'s `open-next` uses the same secret so staged PRs show checks too. The old browser **dispatch mod-management workflows (add-mod / remove-mod / update-mods / compute-hash / refresh) were retired** — the editor does all of it now (#127).
- `build.yml` builds the Prism installer zip (the `pack.toml` URL baked in at build time) and attaches it to a release. It's a **reusable workflow** (`workflow_call`, invoked by `release.yml`), and also runs on a hand-made `release:published` or manual `workflow_dispatch` (test build, no upload). No caching — the build is ~30s. (No `.mrpack`: it bloated to ~200 MB and nobody used it — dropped per #73.)
- `release.yml` **auto-cuts a release** on push to `main` (i.e. when a version PR merges). It reads `version` from `pack.toml` and, **only if tag `vX.Y.Z` doesn't already exist**, calls `build.yml` to create the tag + release with the zip attached. The tag-exists guard makes it idempotent — ordinary main pushes (docs, the `sync-index` `[skip ci]` auto-commits) don't change the version, so they're no-ops. **Version-first**: `pack.toml` drives the tag, so the old "sync pack.toml from the tag" step in `build.yml` was removed.
- `build-editor.yml` runs on `tools/editor-src/**` changes (see above).
- Cutting a release: **bump `version` in `pack.toml` (+ a `docs/PATCHNOTES.md` entry) in the PR → merge to `main`**. `release.yml` tags `v0.X.Y` and publishes the assets in ~1 min. No manual tagging. (Forgetting the bump = no release; the version is the trigger, so don't miss it.)
- **The GitHub release body is pulled from `docs/PATCHNOTES.md`** — `build.yml` extracts the `## X.Y.Z …` section for the version being released (up to the next `## `). So **keep `PATCHNOTES.md` current on the release branch for any change you make there**: whatever that section says when the branch merges is what ships as the public release notes. If it's missing or still the auto-staged `— TBD` stub, the release still cuts but its body is a "notes not finalized" placeholder (and a CI warning fires) — so the failure mode is ugly notes, never a blocked release.
- **Every issue maps to a milestone**, on an **odd/even release cadence** (`docs/RELEASE-CADENCE.md`): **odd = feature/integration + the curation "thunderdome"** (curation by merit), **even = stabilization** (perf/bug/balance, *no new mods*). One pillar per odd version — `v0.7.0` Create spine → `v0.9.0` Economy → `v0.11.0` Survival & Magic → `v0.13.0` Aeronautics → `v0.15.0` Polish + weave review — each trailed by a stabilization even; `v0.6.0` (Foundation) and `v1.0.0` (even, feature-frozen perf patch) bookend, plus `📋 Backlog / living`. The thunderdome runs on the **odd** versions; perf-pruning on the evens. Each milestone carries a one-line **goal** (its exit criterion); the full map is in `docs/ROADMAP.md` ("Milestones — the road to 1.0"). **Any new issue must be filed under a milestone** — its pillar's odd version, the stabilization even that follows, or `Backlog / living` for ongoing/not-yet-scheduled work (the backlog is expected to grow well past today's count, so it's the holding pen that keeps everything triaged). An unmilestoned issue is untriaged. Milestones are created/edited on GitHub, not in this repo.
  - **Close issues on *stage*, not on merge** (`docs/TRIAGE.md` §5): close an issue as `completed` when its fix lands on the version branch, so the milestone bar climbs *as the version is built* — that bar is the at-a-glance "how much of this version is done?" gauge. Keep `playtest-blocked` on the closed issue until it's verified in-game (drop it once playtest passes; a failed playtest reopens). Milestone 100% = all work *implemented*; the hard "shipped" signal is the `vX.Y.Z` release tag (`release.yml`), i.e. 100% **and** no `playtest-blocked` left **and** the tag cut.
  - **Milestone *numbers*** (the GitHub API needs the number, not the title, when filing/moving an issue — so they're recorded here to save a lookup; verify against `/milestones` if a mismatch is suspected): **10** = `v0.6.0 — Foundation`, **11** = `v0.7.0 — Create spine`, **12** = `v0.8.0 — Stabilization I`, **13** = `v0.9.0 — Economy`, **14** = `v0.10.0 — Stabilization II`, **15** = `v0.11.0 — Survival & Magic`, **16** = `v0.12.0 — Stabilization III`, **17** = `v0.13.0 — Aeronautics & logistics`, **18** = `v0.14.0 — Stabilization IV`, **19** = `v0.15.0 — Polish & site`, **20** = `v1.0.0 — Release [NFR]`, **21** = `📋 Backlog / living`.

## Mod / gameplay gotchas

- `aeronauticscompat` (CurseForge) is **required, not optional** — it patches third-party mods to behave on physics-simulated ships (Etched, Cobblemon, turrets, sleeping, chains, …). Keep it in the pack.
- **Recipe viewer is JEI, not EMI.** EMI **cannot render Create's processing recipes** (pressing/mixing/deploying/sequenced-assembly) on 1.21.1 — that hid the entire Create spine (even `create:iron_sheet` showed "no recipe"), which is fatal for a Create pack. Switched to **JEI** (+ **JER** for loot/worldgen/trades, **JEED** for effects) in the recipe-overhaul work. **Don't re-add EMI.**
- **`galosphere:silver_ingot` is actually PALLADIUM** — Galosphere renamed its metal silver→palladium but kept the legacy item *id*. It's tagged `c:ingots/palladium` (its real material), **not** `c:ingots/silver`; the real silver is `occultism`. ⚠️ **Never unify `galosphere:*silver*` as silver** (e.g. an almost-unified `priority_override` on `c:ingots/silver` → galosphere is wrong, and would silently merge palladium into silver). Watch for this on auto-updates that re-touch unification.
- `itemphysic` and `FoamFix` are **client-only — never on the server** (they crash it).
- **Don't add Sodium/Embeddium — they break Create Aeronautics.** Sodium 0.6.9+ on NeoForge 1.21.1 turns **all world textures invisible** when Create: Aeronautics is present (Sodium [#3606](https://github.com/CaffeineMC/sodium/issues/3606), open/unfixed; no working `sodium-mixins.properties` or render-mode workaround). Aeronautics is **required** (so is `aeronauticscompat`), and it hard-depends on **Sable → Veil**, so Veil is **already in the pack by necessity** and Sodium is **off the table** until upstream fixes it. (Embeddium is *incompatible with Veil* outright.) ⚠️ Note the F3 line `[Fabric] Active renderer: IndigoRenderer` = **no chunk-renderer optimizer active** — that's expected here, not a bug to "fix" by adding Sodium. Client FPS is pursued instead via renderer-agnostic mods (already shipped: FerriteCore, MoreCulling, ModernFix, Lithium). Earlier docs wrongly claimed "the pack uses Sodium proper" — it never did and can't. See #98. ⚠️ **#98 update:** the spawn-FPS case is **CPU-bound, not GPU-bound** — F3 showed ~37 FPS with `GPU: 49%` (GPU half-idle), so *draw-side* mods don't help (ImmediatelyFast + EntityCulling were trialed and **reverted — no measured gain**). Suspects are GC pressure (heap was pinned at 100% of 12 GB) and CPU render-thread cost (Flywheel/Veil instance-building for Aeronautics ships); diagnose with `spark profiler --thread "Render thread"` + `spark gc` before adding anything.
- **Iris/Oculus shaders break Aeronautics ships visually** — known upstream bug, no fix. If ships look wrong, the answer is "disable shaders," not debugging shader compat.
- **Boot-error cascades: the first error is the real one.** The rest are dependents failing because their dependency failed; fix the first and the cascade usually clears. Missing-dependency errors are usually packwiz not auto-pulling a dep that isn't jar-in-jar — add it manually.

## Server deploy seam (reference — runs on the box, not here)

The server uses `TYPE=NEOFORGE` and pulls mods from the live `pack.toml` via packwiz-installer at startup, so "deploy a new version" = restart the container, which re-pulls. The only repo-side server task is the NeoForge `[versions]` bump above. Performance (TPS) is profiled with **spark** on the live server — and **measure, don't analogize**: old-pack tuning lessons (e.g. NTNH's 12G-vs-16G) were 1.7.10 / Java 8 and don't carry to 1.21.1 / Java 21 / G1GC.

## Working style

Keep changes focused and explain them briefly. Before changing a mod's presence or a config's values, check `docs/DESIGN.md` (and `mods/README.md` for manifest conventions) — a lot of in/out decisions are deliberate and documented.

- **Player-facing copy — chiefly `docs/PATCHNOTES.md`, plus site text — reads professionally**, the way a person would write it: no decorative emoji, sparing bold, plain declarative prose, and no hype or filler ("the big one", "you'll notice immediately", "seamlessly", "actually"). Internal docs (handoffs, ROADMAP, RECIPES, CLAUDE.md) and commit/PR text **keep their normal heavier formatting** — the bold/structure aids parsing; this professional-prose rule is for player-facing copy only.
